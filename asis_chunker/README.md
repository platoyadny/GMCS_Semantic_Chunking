# AS-IS Document Chunker

Скрипт для чанкинга документов обследования бизнес-процессов AS-IS.
Подготавливает чанки для последующей векторизации и маппинга на банк функциональности PLM/ERP.

## Быстрый старт

```bash
# Базовый запуск (без LLM, все чанки формируются программно)
python chunk_asis.py --input БП_8.docx

# С указанием выходного файла
python chunk_asis.py --input БП_8.docx --output output/chunks_bp08.json

# С LLM-переформулировкой step_enriched чанков (GPT-4o)
cp .env.example .env   # заполнить OPENAI_API_KEY
python chunk_asis.py --input БП_8.docx --use-llm
```

## Зависимости

**Без `--use-llm`:** нет внешних зависимостей, только стандартная библиотека Python 3.10+.

**С `--use-llm`:**
```bash
pip install langchain-openai python-dotenv
```

## Подготовка документа

Документ .docx должен содержать Word Heading стили:

```
[H1] БП 06. Производство продукции              ← 1 на файл

[H2] БП 06.01 Планирование производства на год   ← начало каждого БП
  [H3] Карточка процесса                          ← опциональный
  [H3] Описание хода процесса                     ← обязательный
  [H3] Описание шагов процесса                    ← обязательный (перед таблицей)
  [H3] Особенности реализации AS-IS               ← обязательный
  [H3] Рекомендации                                ← опциональный

[H2] БП 06.02 Планирование на квартал
  ...
```

**Правила разметки:**
- H2 обязательно содержит паттерн `БП XX.YY Название` — из него извлекается ID процесса
- H3 названия могут варьироваться — скрипт классифицирует по ключевым словам в названии,
  а при нестандартном названии — по содержимому секции (fallback)
- Секции могут отсутствовать — скрипт генерирует 0 чанков для отсутствующей секции
- Одна H3 «Рекомендации» может содержать подразделы «по автоматизации» и «по ведению» —
  подтипы определяются из текста автоматически

## Стратегия чанкинга

### Таблица шагов → до 3 чанков на строку

Для каждой строки таблицы создаётся до 3 чанков с разным фокусом:

| Чанк | chunk_type | search_text | Назначение |
|------|-----------|-------------|------------|
| **A** | `step_table` | Название операции + контекст БП | Точный BM25 по ключевым словам |
| **B** | `step_narrative` | Абзац нарратива (если привязка найдена) | Семантический поиск |
| **C** | `step_enriched` | Операция + исполнитель + вход/выход + система | Маппинг по ролям и документам |

**Чанк B** создаётся **только** если в секции «Описание хода процесса» найден абзац
с явной цифровой ссылкой `XX.YY.ZZ` на этот шаг. Никаких семантических догадок —
привязка строго по номеру.

**Чанк C** с флагом `--use-llm` переформулируется GPT-4o в одно связное предложение.
Без LLM — программная конкатенация полей.

Пример трёх чанков для шага «Контроль за проведением ШМР»:

```
A (step_table):
  "В рамках процесса «Реализация договора СС (ШМР, ПНР)»:
   Контроль за проведением ШМР"

B (step_narrative) — только если привязка найдена:
  "Шеф-инженер осуществляет технический надзор за выполнением
   шеф-монтажных работ, ведёт еженедельные отчёты..."

C (step_enriched) — программно:
  "Контроль за проведением ШМР. Исполнитель: Шеф-инженер.
   Вход: Распоряжение Ф11. Выход: Еженедельный отчет Ф14.
   Система: Directum"

C (step_enriched) — с LLM:
  "Шеф-инженер контролирует проведение шеф-монтажных работ (ШМР),
   на основании распоряжения формирует еженедельный отчёт, табель
   рабочего времени и журнал ШМР и ПНР в системе Directum."
```

Все 3 чанка связаны через `pair_id` для дедупликации при поиске.

### Рекомендации → один пункт = один чанк

Одна мысль = всё между двумя цифрами нумерации. Поддерживаются форматы:
- Плоская нумерация: `1. Текст`, `2. Текст`
- Иерархическая: `1.1 Текст`, `2.3 Текст`
- Word-нумерация (цифра в стиле, не в тексте)

Пары «N. Описание проблемы» + «Рекомендуется: решение» склеиваются в один чанк.
В `search_text` попадает **только функциональная часть** (после «Рекомендуется:»),
без описания проблемы — для лучшего совпадения с банком функциональности.

Подтипы определяются автоматически:
- `rec_automation` (priority: high) — «автоматизация»
- `rec_process` (priority: medium) — «ведение процесса», «регламент», «KPI»
- `rec_consulting` (priority: medium) — «консалтинг»

### Остальные секции

| Секция | Чанкование | mapping_priority |
|--------|-----------|-----------------|
| Карточка процесса | 1 чанк целиком | medium |
| Описание хода (непривязанные абзацы) | По абзацам | low |
| Особенности AS-IS | 1 чанк целиком | low |
| Intro (текст до первого H3) | 1 чанк целиком | low |

## Структура выходного JSON

```json
[
  {
    "chunk_id": "BP-08.04.08-table",
    "search_text": "В рамках процесса «...»: Контроль за проведением ШМР",
    "full_context": "Шаг 8: БП 08.04.08...\nИсполнитель: Шеф-инженер\n...",
    "metadata": {
      "bp_id": "08.04",
      "bp_name": "Реализация договора СС (ШМР, ПНР)",
      "chunk_type": "step_table",
      "mapping_priority": "high",
      "step_id": "08.04.08",
      "pair_id": "BP-08.04.08",
      "executor": "Шеф-инженер",
      "system": "Directum",
      "input_docs": "Распоряжение Ф11...",
      "output_docs": "Еженедельный отчет Ф14...",
      "source_doc": "БП_8.docx"
    }
  }
]
```

**Поля:**
- `search_text` — для BM25-индекса и embedding (чистый, без метаданных)
- `full_context` — для LLM и формирования ФТТ (все данные)
- `metadata.pair_id` — связь между чанками A/B/C одного шага (для дедупликации)
- `metadata.mapping_priority` — high / medium / low

## Результаты на тестовых документах

| Документ | БП | Чанков | step_table | step_enriched | step_narrative | rec | narrative |
|----------|-----|--------|-----------|---------------|----------------|-----|-----------|
| БП_08 Управление сервисом | 5 | 160 | 39 | 39 | 12 | 21 | 34 |
| БП_06 Производство | 14 | 696 | 124 | 124 | 123 | 275 | 20 |
| БП_09 Управление ИТ | 4 | 86 | 29 | 29 | 0 | 8 | 8 |

**step_narrative:** БП_06 — 123 из 124 шагов привязаны (нарратив с явной нумерацией).
БП_08 — 12 из 39 (частичная нумерация). БП_09 — 0 (сплошной нарратив).

## Следующий шаг: векторизация и RAG

После чанкинга JSON-файлы используются в RAG-пайплайне:
1. `search_text` → BM25-индекс (OpenSearch) + embedding (BGE-M3) → гибридный поиск
2. Маппинг: AS-IS чанки ↔ банк функциональности (поиск в обе стороны)
3. `full_context` + `metadata` → формирование ФТТ через LLM
